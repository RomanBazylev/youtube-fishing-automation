## Медиа и TTS: стратегия и формат данных

Этот файл описывает:

- откуда берутся кадры (фото/видео) для роликов;
- как хранить эти файлы;
- как n8n и скрипт `video_renderer.py` договариваются о том, какие кадры использовать;
- какой формат запроса/ответа ожидается от TTS‑сервиса.

### 1. Источник медиаконтента

Базовый вариант — **локальная библиотека медиафайлов** в структуре папок:

```text
assets/
  images/
    river/
    fish/
    lures/
    boat/
    sunset/
  video/           # (опционально) короткие клипы вместо картинок
    broll_river/
    broll_boat/
```

- в каждую папку складываются соответствующие изображения/клипы;
- категории (имена папок) должны совпадать с теми, которые используются в `image_hint` и в кадрах `frames` (см. ниже).

### 2. Формат кадров `frames` для `video_renderer.py`

Нода `Prepare Assets JSON` в n8n формирует массив `frames`, который сохраняется в файл `runtime/assets_frames.json`.

Пример содержимого:

```json
[
  {
    "type": "image",
    "category": "river",
    "duration": 3,
    "text": "Сделай первый заброс на глубокой бровке..."
  },
  {
    "type": "image",
    "category": "fish",
    "duration": 3,
    "text": "Не торопись с подсечкой, дай щуке заглотить приманку..."
  },
  {
    "type": "image",
    "category": "sunset",
    "duration": 3,
    "text": "Подпишись, если любишь рыбалку на рассвете!"
  }
]
```

**Интерпретация в `video_renderer.py`:**

- `type` — пока используется значение `image` (картинки); в будущем можно добавить `video`;
- `category` — имя папки внутри `assets/images`, откуда случайно выбирается файл;
- `duration` — желаемая длительность кадра в секундах (будет подстроена под длину озвучки);
- `text` — подпись/субтитр на экране, синхронизированная с этим кадром.

### 3. TTS‑сервис: формат запроса/ответа

Нода `Call TTS` в воркфлоу ожидает HTTP‑API со следующим поведением:

- **Метод**: `POST`
- **URL**: `https://your-tts-endpoint.example.com/tts` (замените на реальный)
- **Формат запроса** (JSON):

```json
{
  "text": "Полный текст озвучки, собранный из сценария"
}
```

- **Формат ответа**:
  - HTTP‑код 200;
  - бинарное тело ответа — аудиофайл (`audio/mpeg` или `audio/wav`).

В настройках ноды `Call TTS`:

- `Response Format` — `File`;
- далее бинарный объект с озвучкой передаётся в ноду `Execute Video Renderer` как `audio-path` (см. вызов в `n8n_workflow_youtube_fishing_generator.json` — при необходимости поправьте под конкретный API).

### 4. Варианты TTS‑провайдеров

На практике вы можете использовать:

- коммерческий сервис (например, ElevenLabs, Яндекс SpeechKit, GCP Text‑to‑Speech и т.п.);
- локальный TTS‑сервер, развернутый у вас (например, через Docker).

Требования:

- должен уметь принимать текст на русском;
- должен возвращать аудиофайл в одном из популярных форматов (`mp3` или `wav`).

При подключении конкретного сервиса:

- замените URL в ноде `Call TTS`;
- адаптируйте тело запроса (`bodyParametersJson`) и заголовки авторизации;
- при необходимости скорректируйте `video_renderer.py`, если используется специфичный формат (например, только `wav`).
